Standard Operating Procedure

1

Enter client name, site, engineer and revision. These stamp every export so documents are traceable. Set the protocol and PLC IP — the address column will show the correct format per protocol.

2

Load or build the map

Click Load Standard Template for a ready-made PLC→Robot→CNC layout. Customize groups and registers. For a new client, load template then add/delete rows to match the actual PLC program.

3

Complete every value map

Click + value on any register to add what each number means — 0 = Home, 1 = Pallet Pick, etc. Every position/status register must be fully mapped. Undocumented values cause wrong behavior after power restore.

4

Power-cut recovery registers

POWER_RESTORE_STATE stores the robot's last safe position before any move. After power comes back, the HMI operator sets HMI_HOME_CMD = 1. PLC reads the saved state and issues the correct move command.

5

Address numbering rules

For Modbus: use 40001-series for all command/status registers (holding registers, R/W 16-bit). Use 30001-series for read-only sensors. Use 00001-series coils for single-bit discrete outputs. See the Protocol Deep Dive below for other protocols.

6

Export and version control

Export JSON for version control (Git, SharePoint). Export CSV for client documentation. Print/PDF for site commissioning binder. File naming: RegisterMap_<ClientCode>_R<Rev>.json

IEC 61158 · Most common PLC↔Robot protocol · Port 502 (TCP)

The Modbus Data Model — 4 separate tables

Modbus defines four distinct address spaces, each with its own purpose. The first digit (or prefix) tells you which table you're in:

0x Coils
1x D.Input
3x In.Regs
4x Holding Regs ← most used
PrefixRangeTypeAccessFC (read)FC (write)Best used for
0x00001–09999Coil — 1 bitR/WFC 01FC 05/15Discrete output bits (relays, solenoids)
1x10001–19999Discrete Input — 1 bitR onlyFC 02—Digital sensor inputs
3x30001–39999Input Register — 16-bitR onlyFC 04—Analog sensor values, encoder counts
4x40001–49999Holding Register — 16-bitR/WFC 03FC 06/16Commands, status, parameters — all comms

Why does 40001 actually mean address 0 on the wire?

The "40001" notation is a human-readable reference number, not the actual Modbus PDU address. The Modbus standard is zero-based on the wire:

Reference 40001 → PDU address 0x0000 (zero)
Reference 40002 → PDU address 0x0001
Reference 40100 → PDU address 0x0063

So when programming your Modbus master (PLC), subtract 40001 from the reference number to get the actual register address to put in your FC03/FC16 request. Most HMI and PLC software does this automatically.

Always confirm with the robot/CNC supplier whether they use 1-based (40001 = register 1) or 0-based (40001 = register 0) addressing in their documentation. This single mismatch causes the most commissioning errors.

Why use 40001-series for robot/CNC comms?

Holding registers (4x) are used for all PLC↔Robot↔CNC communication because they are:

• Bidirectional — PLC can read (FC03) and write (FC06/16)
• 16-bit wide — holds integers, status words, command codes
• Pairable — two consecutive registers = 32-bit DWORD or IEEE 754 float
• Universally supported — every Modbus device implements FC03/FC16
• Block readable — FC16 can write up to 123 registers in one request

No register numbers — it uses named tags

EtherNet/IP uses the Common Industrial Protocol (CIP). Instead of numeric register addresses, you address data by tag name — a symbolic string that maps to a memory location in the controller. This is fundamentally different from Modbus.

Implicit vs Explicit messaging

Implicit (I/O) messaging — cyclic, real-time. A "connection" is established and data is exchanged every scan cycle automatically. Used for high-speed status/command exchange between PLC and robot. Assembly objects define the data layout.

Explicit messaging — request/response, on-demand. PLC sends a CIP read/write request; device responds. Used for parameter reads, recipe uploads, alarm retrieval. Slower but flexible.

For robot comms, use Implicit messaging with the robot vendor's I/O assembly objects. The vendor's manual will define Output Assembly instance (PLC writes → Robot) and Input Assembly instance (Robot writes → PLC reads).

CIP path structure

When connecting to a device, you specify a CIP path: 1,0 means backplane, slot 0. 1,0,1,1 means backplane slot 0, then forward to slot 1. For direct Ethernet devices (like a FANUC robot), the path is typically just the IP address — no backplane path needed.

PROFINET — Slot/Subslot/Byte model
IEC 61158 · Siemens-led · Real-time classes RT, IRT · Port 34964
▼

How PROFINET addresses data

PROFINET does not use register numbers or tag names. Data is addressed using a slot / subslot / byte offset model defined by the device's GSDML file. The GSDML (device descriptor) tells the PLC what modules (slots) the device has and what data is at each offset.

ConceptDescriptionExample
SlotPhysical or virtual module positionSlot 1 = Robot I/O module
SubslotSub-position within a slotSubslot 1 = first submodule
Byte offsetByte position in the I/O data areaOffset 0 = first byte
Process ImagePLC memory area mapped from PROFINET I/OIB0, QB0 in Siemens
GSDMLXML descriptor file from device vendorImported into TIA Portal / STEP 7

Output (PLC→Robot) vs Input (Robot→PLC)

In PROFINET, Output means data flowing from the PLC controller to the field device (robot). Input means data flowing from the field device back to the PLC. This is opposite to how you'd describe it from the robot's perspective.

A typical robot PROFINET I-Device (Intelligent Device) layout:

DirectionPLC viewRobot viewContent
OutputWrites to QB0…QB15Reads from input areaCommand word, mode, setpoints
InputReads from IB0…IB15Writes to output areaStatus word, position, alarms

PROFINET real-time classes

RT (Real-Time) — standard, cycle times 1–512 ms. Sufficient for robot command/status. Uses Ethernet multicast, no special hardware needed.

IRT (Isochronous Real-Time) — cycle times down to 250 µs. Required for motion synchronization (e.g. robot path following CNC axis). Requires IRT-capable switches (e.g. Siemens SCALANCE X).

For simple pick-and-place automation, RT is sufficient. If the robot is doing synchronized axis motion with CNC (e.g. in-process loading), specify IRT from the start — adding it later requires hardware changes.

PROFINET device naming

Every PROFINET device has a Device Name (ASCII, e.g. fanuc-robot-01) that must be unique on the network segment. The PLC assigns the device name and IP address during startup using DCP (Discovery and Configuration Protocol). The name is stored in the device's non-volatile memory.

What makes OPC-UA different

OPC-UA is not a fieldbus — it is a data modeling and transport protocol designed for vertical integration (PLC → SCADA → MES → ERP). It is used alongside a fieldbus (Modbus, PROFINET, etc.), not instead of it. The PLC exposes its data as OPC-UA nodes, and upper systems subscribe to them.

LayerProtocolPurpose
Field levelPROFINET / Modbus / EIPPLC ↔ Robot ↔ CNC real-time exchange
Control levelOPC-UA server on PLCPLC exposes data to SCADA / HMI
MES / ERP levelOPC-UA clientReads cycle counts, faults, KPIs

Node ID formats

Every data point in OPC-UA has a Node ID consisting of a namespace index and an identifier:

FormatSyntaxExample
Numericns=<ns>;i=<id>ns=2;i=1001
Stringns=<ns>;s=<name>ns=2;s=Robot.Position
Browse path/Objects/PLC/Robot/PositionHuman-readable path in node tree

Namespace 0 is reserved by OPC Foundation. Your PLC data lives in namespace 2 (or higher). The numeric ID maps to the register/tag in the PLC — the PLC's OPC-UA server defines the mapping.

Subscriptions vs polling

OPC-UA clients subscribe to nodes with a publishing interval (e.g. 500 ms). The server only sends data when it changes (deadband filtering). This is far more efficient than Modbus polling where you read even if nothing changed.

OPC-UA over TSN (Time-Sensitive Networking) is the emerging standard for combining OPC-UA with real-time control, potentially replacing PROFINET and EtherNet/IP in future installations.

Security in OPC-UA

OPC-UA has built-in security: authentication (username/password or certificates), signing (data integrity), and encryption (confidentiality). For SCADA integration, always use at minimum Sign & Encrypt mode with certificate-based authentication. This is mandatory for IEC 62443 compliance.

CC-Link IE Field — Link registers (W) and relays (B)
Mitsubishi-led · CLPA standard · 1 Gbps · Common in Mitsubishi / FANUC / Yaskawa lines
▼

Address spaces in CC-Link IE

CC-Link IE Field uses a memory-mapped approach with two main address spaces — link relays (bit data) and link registers (word data). Addresses are in hexadecimal:

TypeRangeWidthAccessUse
RX / RYRX0000–RXFFFF1 bitRX: slave→master, RY: master→slaveDigital I/O, handshake bits
RWr / RWwRWr0000–RWrFFFF16-bit wordRWr: slave→master, RWw: master→slaveStatus, commands, parameters
BB0000–BFFFF1 bitShared across all stationsGlobal signals, interlocks
WW0000–WFFFF16-bit wordShared across all stationsData exchange between stations

Station numbers

Each device (robot, CNC, drives) has a station number (1–120). The master PLC (station 0) allocates a block of RX/RY/RWr/RWw addresses per station based on the configured point count. Each station's data occupies a contiguous, non-overlapping block.

Example allocation with 32 points per station:

StationDeviceRY (out)RX (in)RWw (out)RWr (in)
1FANUC RobotRY0000–RY001FRX0000–RX001FRWw0–RWw7RWr0–RWr7
2Fanuc CNCRY0020–RY003FRX0020–RX003FRWw8–RWwFRWr8–RWrF

Cyclic vs transient transmission

Cyclic — automatic, every scan cycle. RX/RY/RWr/RWw are exchanged without any explicit request. The PLC always has fresh data. This is equivalent to Modbus polling but automatic.

Transient — on demand, using SEND/RECV instructions. Used for recipe uploads, parameter changes, alarm history retrieval.

CC-Link IE is dominant in Japanese OEM equipment (Fanuc, Yaskawa, Mitsubishi, DMG Mori). If the robot or CNC is Japanese-made, check whether CC-Link IE is already available as an option before choosing Modbus — it will give faster cycle times and tighter integration.

PR
Power Restore & Homing Logic — How it works
Critical safety sequence · Applies to all protocols · Must be programmed in PLC
▼

The problem: robot could be anywhere when power cuts

When power is cut — whether from an emergency stop, mains failure, or planned shutdown — the robot may stop mid-trajectory. On power restore, the PLC and HMI have no idea where the robot is physically unless a register was tracking this. Sending a "go to position" command without knowing where the robot is can cause a collision.

The register solution

RegisterModbus addrWritten byWhen written
POWER_RESTORE_STATE40404PLCAfter every confirmed move — stores last COMPLETED position index
ROBOT_POS40001RobotEvery scan — current position (may be mid-move, unreliable after e-stop)
HMI_HOME_CMD40403HMI operatorManually set after power restore, after verifying it is safe to home
POWER_RESTORE_FLAG40103PLC (on boot)Automatically set by PLC on startup if previous state was not "clean stop"

Homing sequence — step by step

Step 1 — Power restores: PLC boots. It reads POWER_RESTORE_STATE from retentive memory (battery-backed or FRAM). If it is not 0 (Home), it sets POWER_RESTORE_FLAG = 1.

Step 2 — HMI shows warning: HMI detects POWER_RESTORE_FLAG = 1 and shows "Last known position: [Transition — Pallet→CNC]. Robot must home before running. Confirm it is safe, then press HOME."

Step 3 — Operator confirms and presses HOME: Operator visually confirms robot position, verifies no obstruction. Presses HOME on HMI. HMI sets HMI_HOME_CMD = 1.

Step 4 — PLC issues command: PLC checks POWER_RESTORE_STATE. If it was at a CNC position, PLC first issues CMD_ROBOT = 6 (go CNC unload path), waits for ACK, then issues CMD_ROBOT = 1 (go home). If it was at pallet, it goes directly home. The path depends on the saved state.

Step 5 — Robot confirms home: Robot sets ROBOT_POS = 0. PLC clears POWER_RESTORE_FLAG, sets POWER_RESTORE_STATE = 1 (clean/at home). System is ready for auto mode.

Never write POWER_RESTORE_STATE speculatively or before the move is confirmed complete. Write it only when the robot's ACK register confirms the position. Writing it during a move gives a false safe state.
Use retentive (battery-backed) PLC memory for POWER_RESTORE_STATE. On Siemens S7 this is a DB with "Non-optimized" and "Retain" flags. On Allen-Bradley this is Retain attribute on the tag. On Mitsubishi this is file register R or battery-backed D range.
